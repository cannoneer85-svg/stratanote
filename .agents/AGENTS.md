# Obsidian Collab - Правила для ИИ-агентов (AGENTS.md)

Эти правила обязательны к исполнению всеми ИИ-агентами в данном рабочем пространстве.

## 1. Запрет на автоматические Git-операции

*   **КРИТИЧЕСКИ ВАЖНО**: Никогда не выполняйте `git commit` или `git push` автоматически в фоновом режиме или в рамках обычных задач без прямой команды или подтверждения от пользователя.
*   Все изменения исходного кода должны оставаться в рабочем каталоге для проверки пользователем.
*   Для коммитов и пушей используйте только соответствующие слэш-командные навыки `/commit` и `/push`, которые сначала запрашивают подтверждение изменений у пользователя.
*   **ОБЯЗАТЕЛЬНОЕ обновление статуса проекта**: При выполнении каждого коммита (через `/commit` или вручную) ИИ-агент обязан обновить файл `_app/project_status.md` с целью поддержания его в актуальном состоянии. В файле должны быть зафиксированы все ключевые изменения кода (новые функции, багфиксы, изменения структуры или архитектуры), выполненные в рамках коммита. Данный файл должен всегда точно отражать текущее состояние проекта.
*   **ОДНОРАЗОВОСТЬ ПОДТВЕРЖДЕНИЙ**: Каждая операция `git commit` и `git push` требует отдельной явной команды (например, `/commit` и `/push`) или отдельного подтверждения от пользователя. Запрещено выполнять автоматические коммиты и пуши в рамках «продолжения отладки», «горячих исправлений» (hotfix) или устранения ошибок сборки, даже если предыдущие шаги в рамках того же диалога были одобрены пользователем. Любые новые изменения кода должны быть сначала продемонстрированы пользователю и согласованы отдельно.

## 2. Стандарты именования коммитов (Conventional Commits)

Все коммиты должны строго следовать формату Conventional Commits:
`тип(область): Краткое описание на английском`

Допустимые типы:
*   `feat`: Новая функциональность (вызывает минорное повышение версии MINOR).
*   `fix`: Исправление ошибки (вызывает патч-повышение версии PATCH).
*   `docs`: Изменения в документации.
*   `style`: Код-стиль, форматирование (без изменения бизнес-логики).
*   `ref`: Рефакторинг кода (без изменения поведения).
*   `perf`: Улучшение производительности.
*   `test`: Добавление или исправление тестов.
*   `chore`: Обслуживание сборки, зависимостей и т.д.

Если изменение ломает обратную совместимость (BREAKING CHANGE), в заголовке после типа должен стоять знак восклицания `!`, либо это должно быть указано в футере коммита (вызывает мажорное повышение версии MAJOR).

## 3. Версионирование проекта (SemVer)

При подготовке релиза с помощью команды `/release` необходимо:
1.  Следовать стандарту **Semantic Versioning 2.0.0** (`MAJOR.MINOR.PATCH`).
2.  Автоматически анализировать изменения со времени последнего релиза (тега).
3.  Определять необходимый инкремент (MAJOR, MINOR или PATCH).
4.  Составлять описание изменений (Keynotes) на русском языке для отображения пользователям в интерфейсе приложения.
5.  Обновлять файлы версий проекта с помощью скрипта `prepare-release.js`.

## 4. Контекст проекта (Project Context)

*   **ОБЯЗАТЕЛЬНО**: При начале любого нового диалога или перед выполнением первой задачи обязательно прочитайте файл `_app/project_status.md` с помощью инструмента `view_file`. Это необходимо для того, чтобы всегда быть в курсе текущего статуса проекта, его архитектуры и последних изменений. Не приступайте к изменениям кода, пока не ознакомитесь с этим файлом.
