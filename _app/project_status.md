# Статус проекта и архитектура — StrataNote

**StrataNote** — это веб-приложение для совместной работы с базой знаний на основе Markdown. Проект сочетает визуальный опыт классического Obsidian (двусторонние вики-ссылки, граф связей, папки и файлы) с возможностями многопользовательского редактирования в реальном времени, контролем версий и администрированием.

---

## 1. Стек технологий и архитектура

Система разделена на три ключевых слоя:

### Клиент (Frontend)
Расположен в каталоге [_app/client](file:///d:/YandexDisk/AWG/md_obsidian/_app/client). Built на React + TypeScript + Vite.
* **Редактор**: построен на базе CodeMirror 6 с кастомными расширениями для подсветки синтаксиса Markdown и автодополнения вики-ссылок `[[заметка]]` при вводе.
* **Граф связей**: интерактивная 2D-визуализация сети документов на базе Force-directed graph (D3.js / HTML5 Canvas).
* **Стилизация**: CSS-переменные с поддержкой премиального темного режима, эффектов стекла (Glassmorphism), размытия фона и неонового свечения.

### Сервер (Backend)
Расположен в каталоге [_app/server](file:///d:/YandexDisk/AWG/md_obsidian/_app/server). Написан на Node.js + Express.
* **База данных**: SQLite (файл `database.sqlite` инициализируется через [db.js](file:///d:/YandexDisk/AWG/md_obsidian/_app/server/db.js)). Хранит пользователей, роли, историю версий файлов и активные блокировки.
* **Синхронизация в реальном времени**: Socket.io. Обеспечивает мгновенный обмен присутствием пользователей («кто онлайн и что редактирует»), уведомления об изменениях файлов и установку блокировок на редактирование.
* **Файловый наблюдатель (Watcher)**: библиотека Chokidar в [watcher.js](file:///d:/YandexDisk/AWG/md_obsidian/_app/server/watcher.js). Автоматически индексирует изменения, сделанные на диске внешними программами, обновляет БД SQLite и рассылает сокет-события клиентам. Для предотвращения индексации системных файлов и скриптов в watcher встроен механизм фильтрации (игнорирует папки `_app`, `_sync_mcp`, `.agents`, `node_modules`, `.git`, `.obsidian` и `assets`).

### Облачная инфраструктура (Railway)
* **Деплой**: автоматическая сборка контейнера при пуше в ветку `master` на GitHub. В настройках Railway корневой каталог (Root Directory) переключен на `_app`. Скрипт `postinstall` в `_app/package.json` автоматически запускает установку зависимостей для клиента и сервера, а флаг `--omit=none` принудительно подтягивает TypeScript (`tsc`) и Vite в production-окружении для сборки фронтенда.
* **Персистентность**: подключен постоянный диск (Persistent Volume) в точку монтирования `/data` для сохранения базы данных SQLite (`/data/database.sqlite`) и каталога заметок (`/data/vault`) между деплоями и перезапусками контейнеров.

## 2. Ключевые архитектурные решения

* **Блокировка документов (Conflict Prevention)**: чтобы два редактора не перезаписали работу друг друга, при открытии вкладки в редакторе сервер накладывает эксклюзивный `Lock` на файл. Другие пользователи видят плашку «Редактирует: Имя пользователя» и могут просматривать документ только в режиме чтения (ReadOnly). Для предотвращения бесконечных циклов авто-блокировки/разблокировки в сокетах, клиент отфильтровывает собственную блокировку пользователя при передаче пропа `lockedBy` в редактор.
* **Маршрутизация по хэшу (Hash Routing)**: адресная строка браузера отражает путь к текущей открытой заметке в хэш-секции (например, `/#Общая архитектура проекта.md`). Приложение считывает этот хэш не только при кликах, но и при первоначальной загрузке страницы или ручном обновлении (F5), мгновенно открывая нужный документ и восстанавливая вкладки.
* **Двусторонние вики-ссылки (WikiLinks)**: ссылки вида `[[Имя заметки]]` или `[[Путь/Имя|Альтернативный текст]]` парсятся на сервере для построения связей графа и на клиенте для мгновенного перехода между заметками.
* **История версий (Revisions)**: при каждом сохранении файла на сервере контент записывается в таблицу `versions`. Клиенты могут открыть панель истории, выбрать любую прошлую версию, посмотреть визуальный Diff (разницу) в компоненте [DiffViewer.tsx](file:///d:/YandexDisk/AWG/md_obsidian/_app/client/src/components/DiffViewer.tsx) и восстановить её. История версий использует понятную локальную нумерацию (Версия #1, #2...) для каждой заметки отдельно (вместо сквозных глобальных ID), а для текущей активной версии автоматически выводится бейдж «Текущая версия» и скрывается избыточная кнопка восстановления. Для корректности сравнения система сопоставляет выбранную версию N с её непосредственной предыдущей версией N-1 из базы данных (а не с текущей рабочей копией на диске), что позволяет точно видеть изменения, внесенные именно этой конкретной версией.
* **Механизм рецензирования (Suggest Mode)**: режим предложений позволяет пользователям редактировать файлы, даже если они заблокированы другими соавторами. Изменения сохраняются в виде предложений (черновиков) в таблице `suggestions` базы данных. Для принятия правок используется алгоритм построчного трехстороннего слияния (Three-Way Merge), реализованный в [merge.js](file:///d:/YandexDisk/AWG/md_obsidian/_app/server/merge.js). Модерировать (принимать/отклонять) предложения могут только создатели файлов (поле `created_by` в таблице `notes`) и администраторы.
* **Устойчивость к сбоям (Self-Healing)**: сервер сверяет состояние диска и базы данных SQLite. При расхождениях (например, если файл удален с диска вручную) watcher автоматически удаляет запись и связанные версии из базы данных и синхронизирует списки у активных пользователей.

## 3. Реализованный функционал

### Редактор и рендеринг Markdown (`Editor.tsx` + `index.css`)
* **Поддержка рендеринга диаграмм Mermaid**:
  * Интегрировано построчное автоформатирование диаграмм (subgraph, node, edge labels) в кавычки для поддержки спецсимволов (пробелы, скобки, слеши) для блок-схем (`flowchart` / `graph`).
  * Исправлен рендеринг диаграмм сущностей (`erDiagram`), классов (`classDiagram`) и последовательностей (`sequenceDiagram`) путем изоляции построчного форматирования меток (чтобы не повреждать синтаксис отношений вроде `||--||` или `->>`).
  * Добавлена корректная обработка круглых `((текст))` и овальных `([текст])` узлов в блок-схемах с использованием регулярных выражений с негативным просмотром (lookaround).
  * Выборочное декодирование стрелок с двойными наконечниками (`->>`, `-->>`) и связей классов (`<|--`, `<|..`, `..|>`, `..>`).
  * Полное восстановление HTML-сущностей сравнения (`&lt;` -> `<`, `&gt;` -> `>`, `&amp;` -> `&`) для всех типов диаграмм, отличных от блок-схем, для сохранения знаков сравнения в текстах примечаний и подписей (например, в sequenceDiagram).
  * Добавлено автоматическое экранирование двойных кавычек (`&quot;`) и точек с запятой (`&#59;`) в текстах сообщений `sequenceDiagram` для защиты от синтаксических ошибок и разделения команд.
  * Разделена нормализация стрелок с дефисами (теперь применяется только к блок-схемам, предотвращая ошибочную замену сплошных стрелок на пунктирные в `sequenceDiagram`).
  * Исправлен баг генерации ID подграфа (`subgraph`) при объявлении в кавычках: идентификатор теперь автоматически очищается от пробелов и спецсимволов.
  * Автоматическое экранирование обратных слэшей (`\`) внутри текстовых меток.
  * Динамический MutationObserver для асинхронного рендеринга Mermaid-схем без циклического зависания страницы.
  * Элегантное аварийное оформление для упавших или некорректных диаграмм (`.mermaid[data-processed="failed"]`) с сохранением всех переносов строк (`white-space: pre-wrap;`), моноширинного шрифта и подсветки ошибок красным цветом.
  * Добавлена автоматическая горизонтальная прокрутка (scroll) для широких Mermaid-диаграмм в режиме предпросмотра (Preview) с предотвращением нечитаемого масштабирования за счет переопределения `max-width` и `width` для SVG.
  * **Полноэкранный интерактивный режим просмотра (Pan & Zoom)**: клик на любую успешно отрендеренную диаграмму открывает полноэкранное модальное окно. Реализовано перетаскивание (drag-to-pan) и зумирование колесиком мыши с фокусом на курсор, а также кнопки управления масштабом (+/-), центрирования и закрытия. При наведении на диаграммы в режиме просмотра меняется курсор (`zoom-in`), подсвечивается фиолетовая рамка и отображается бейдж-подсказка «🔍 Клик для увеличения». Дополнительно реализована высокая оптимизация рендеринга (прямой обход циклов обновления React-состояния во время перемещения мыши и отключение CSS-анимаций при ручном перетаскивании/зуме для достижения стабильных 60 FPS с сохранением идеальной векторной четкости SVG-элементов при любом уровне масштабирования).
* **Цитаты и Obsidian Callouts (Алерты)**:
  * Построчный группирующий парсер цитат (`> `), поддерживающий рекурсивное форматирование внутреннего контента (списки, ссылки, жирный шрифт) с автоматическим сохранением переносов строк (через генерацию HTML `<br />` для последовательных строк, исключая блоки кода, списки и заголовки).
  * Полная поддержка алертов Obsidian вида `> [!IMPORTANT] Title`.
  * Цветовое кодирование и встроенные SVG-иконки для алертов типов `IMPORTANT`, `NOTE`, `WARNING`, `TIP`, `CAUTION` и др., стилизованные под темную тему.
* **Рендеринг и встраивание**:
  * Поддержка стандартного Markdown (`![Alt](assets/file.png)`) и вики-встраивания Obsidian (`![[file.png]]` или `![[video.mp4|autoplay|loop]]`).
  * Чистый рендеринг вики-ссылок `[[RelativePath]]` and `[[RelativePath|Label]]` в режиме просмотра без скобок.
  * Интегрированный HTML5 видеоплеер с автозапуском, зацикливанием и приглушением звука.
  * Масштабирование картинок через синтаксис размеров Obsidian (например, `![[image.png|300x200]]`).
  * Рендеринг адаптивных Markdown-таблиц с поддержкой выравнивания (`|---|:---:|---:|`) и форматирования внутри ячеек.
  * Поддержка разделительных линий (`---`, `***`, `___`).
  * Сохранение режима Код/Просмотр в `localStorage` (ключ `editor_mode`) при переходах между заметками.
  * **Оптимизация изображений в превью (Sharp Image Preview)**: в режиме просмотра (Preview) все локальные изображения запрашиваются с автоматическим указанием ширины (по умолчанию 800px или по явному указанию в разметке: `![alt|300](path)` / `![[path|300]]`). При наведении на картинку в режиме Preview курсор меняется на лупу (`cursor-zoom-in`), делая возможность увеличения очевидной. В режиме редактирования (Edit) для встроенных превью изображений в CodeMirror также автоматически запрашиваются оптимизированные миниатюры шириной 300px вместо загрузки оригиналов. Дополнительно встроенный просмотр изображений при редактировании поддерживает плавное увеличение высоты до 288px при наведении (hover) с курсором `zoom-in`, а клик по картинке открывает оригинальный файл высокого качества в полноэкранном лайтбоксе (как и в режиме Preview). Сервер на лету генерирует и кэширует сжатые миниатюры с помощью библиотеки `sharp`, существенно снижая нагрузку на сеть, экономя память и ускоряя загрузку страниц.
  * **Полная поддержка заголовков `h1`–`h6`**: Все шесть уровней заголовков Markdown (`#` — `######`) корректно рендерятся в режиме Preview с индивидуальными CSS-стилями (уменьшающийся размер шрифта, контрастность и отступы для визуальной иерархии). Каждый заголовок автоматически генерирует `id`-атрибут из slug-версии текста с полной поддержкой кириллицы (Unicode `\u0400-\u04FF`). Порядок regex-замен идёт от `######` к `#` для предотвращения «жадного» перехвата коротких паттернов.
  * **Внутридокументная навигация по якорям (Anchor Navigation)**: В режиме Preview ссылки вида `[текст](#якорь)` из Table of Contents корректно прокручивают страницу к целевому заголовку. Якорные ссылки (`#...`) остаются на странице (без `target="_blank"`), а внешние ссылки по-прежнему открываются в новой вкладке. Обработчик клика выполняет плавную прокрутку (`scrollIntoView({ behavior: 'smooth' })`) к целевому элементу внутри preview-контейнера.
  * **Одиночные переносы строк (Strict Line Breaks)**: Одиночный перенос строки в Markdown-исходнике рендерится как `<br />` в Preview (поведение Obsidian), а не игнорируется как в стандартном Markdown. Regex с негативным lookbehind/lookahead пропускает переносы рядом с блочными HTML-элементами (`h1`–`h6`, `li`, `div`, `hr`, `table`, `pre` и др.) для предотвращения лишних пустых строк.
  * **Экспорт / скачивание отдельного файла**: Добавлена кнопка «Скачать файл в формате MD» с иконкой `Download` в тулбаре редактора. Она позволяет мгновенно скачать текущий просматриваемый/редактируемый файл на локальный диск в виде `.md` документа. Функция работает на стороне клиента (Blob) и доступна всем пользователям в любом режиме доступа (чтение/редактирование/блокировка).
  * **Иерархические вложенные списки и стилизация WikiLinks**:
    - Реализован автоматический расчет вложенности для всех типов списков (маркированные, нумерованные, списки задач) на основе количества пробелов и табов в начале строки с применением динамического сдвига (`margin-left: ...rem`). Это решает проблему плоского отображения вложенных пунктов в содержании заметок.
    - Добавлено отображение SVG-иконки документа (`lucide-file-text`) перед текстом внутренних WikiLinks `[[ссылка]]` и относительных ссылок на `.md`-файлы. Ссылки по умолчанию используют фиолетовый стиль проекта (`text-primary`) с эффектом подчеркивания при наведении (`hover:underline`).
    - **Сохранение скролла вкладок (Scroll Memory)**: Реализовано автоматическое сохранение высоты прокрутки (`scrollTop`) для каждой открытой заметки в режиме Preview. При переключении вкладок или закрытии соседней вкладки положение скролла возвращенного документа восстанавливается из кэша (с защитой от конфликтов при ручном переключении режима Код/Просмотр).
    - **Горячие клавиши (Hotkeys)**: Реализована обработка сочетания клавиш **Ctrl+S** (и **Cmd+S** на macOS) для мгновенного сохранения изменений. Сочетание перехватывается в редакторе, предотвращает вызов системного диалога сохранения страницы браузера и вызывает сохранение документа (или отправку предложения в режиме Suggest Mode), дублируя действие кнопки «Сохранить».
  * **Адаптация для мобильных устройств**:
    - **Тулбар редактора**: На мобильных экранах тулбар делится на две независимые компактные строки. Верхняя строка (кнопки форматирования) прокручивается горизонтально без переносов кнопок, а нижняя строка содержит режимы Код/Просмотр, скачать, сохранить и статус блокировки/синхронизации.
    - **Сокращение статусов**: Тексты статусов синхронизации и блокировки автоматически сокращаются (например, «Синхронизация активна» -> «Синхронизация», «Режим чтения (ReadOnly)» -> «Чтение»), чтобы вмещаться в экран.
    - **Адаптивные отступы**: Внутренние отступы превью-режима изменены с `p-8` на `p-4 sm:p-8`, чтобы на узких мобильных экранах текст не сжимался.
    - **Иконки навигации**: Текст на кнопках переключения вкладок «Заметки» и «Граф» в набаре скрыт на экранах шириной < 768px (оставлены только иконки `FileText` и `Network`), освобождая место для вкладок.
* **Поддержка локальных медиафайлов**:
  * **Поблочная загрузка медиафайлов (Chunked Media Upload)**: реализована поблочная бинарная загрузка файлов (блоками по 5 МБ) через эндпоинт `POST /api/notes/upload-media-chunk`. Это решает проблему ошибки `HTTP 413: Payload Too Large` и позволяет загружать видео- и медиафайлы размером более 300 МБ непосредственно через редактор (Drag & Drop, Paste, клик на кнопку). Временный каталог `tempDir` перенесен в универсальный скрытый каталог хранилища `vaultPath/.temp` (что защищает от переполнения системного диска/контейнера). Добавлена гарантированная немедленная очистка всех временных файлов при любых ошибках слияния, а также автоматическая фоновая самоочистка (stale cleanup) файлов старше 2 часов в `.temp` каждые 30 минут. На клиенте добавлен размытый оверлей с отображением процента загрузки (0% - 100%) и кнопкой отмены, которая прерывает активный сетевой XHR-запрос и останавливает очередь файлов.
  * **Массовая загрузка медиафайлов (Bulk Media Upload)**: реализована возможность одновременного выбора и загрузки нескольких медиафайлов (через Drag-and-Drop, вставку Ctrl+V или кнопку панели инструментов с множественным выбором `multiple`). Загрузка файлов выполняется последовательно для предотвращения перегрузки канала. Файлы вставляются в текст по очереди по мере окончания загрузки, разделяясь пустой строкой (`\n\n`) для красивого форматирования. Оверлей загрузки отображает детальный прогресс для каждого файла (например, `Загрузка медиафайлов (2/5)... 45%`) и поддерживает мгновенную отмену всей операции.
  * **Исправление фокуса курсора**: при вставке медиафайлов (как одиночных, так и массовых) редактор CodeMirror гарантированно возвращает фокус ввода, а курсор позиционируется в конце вставленной разметки с автопрокруткой на экран. Это позволяет сразу продолжать редактирование (например, нажать Enter для новой строки) без кликов мышкой.
  * **Интерактивное разрешение конфликтов имен**: перед загрузкой медиафайла отправляется запрос к `GET /api/notes/media-exists`. При обнаружении дубликата клиенту выводится современный диалог с предложением перезаписать существующий файл, создать копию (с безопасным суффиксом вида `_1` вместо скобок ` (1)` во избежание повреждения Markdown-ссылок) или отменить загрузку. При массовой загрузке конфликты обрабатываются последовательно для каждого файла, а диалог поддерживает чекбокс *«Применить ко всем оставшимся файлам»* (Apply to all remaining files) для автоматического выбора действия для всего пакета файлов.
  * **Оптимизация изображений в превью (Sharp Image Preview)**: в режиме просмотра (Preview) все локальные изображения запрашиваются с автоматическим указанием ширины (по умолчанию 800px или по явному указанию в разметке: `![alt|300](path)` / `![[path|300]]`). При наведении на картинку в режиме Preview курсор меняется на лупу (`cursor-zoom-in`), делая возможность увеличения очевидной. В режиме редактирования (Edit) для встроенных превью изображений в CodeMirror также автоматически запрашиваются оптимизированные миниатюры шириной 300px вместо загрузки оригиналов. Дополнительно встроенный просмотр изображений при редактировании поддерживает плавное увеличение высоты до 288px при наведении (hover) с курсором `zoom-in`, а клик по картинке открывает оригинальный файл высокого качества в полноэкранном лайтбоксе (как и в режиме Preview). Сервер на лету генерирует и кэширует сжатые миниатюры с помощью библиотеки `sharp`, существенно снижая нагрузку на сеть, экономя память и ускоряя загрузку страниц.
  * **Массовое удаление медиафайлов (Bulk Media Deletion)**: реализован эндпоинт `POST /api/notes/media-bulk-delete` для безопасного удаления группы файлов в один клик. В интерфейсе библиотеки медиафайлов (Settings -> Media) добавлены чекбоксы выбора у каждого файла (отображаются при наведении и выборе), поддержка быстрого выделения всей отфильтрованной сетки файлов («Выбрать все») и кнопка «Удалить выбранные (N)» с единым окном подтверждения.
  * **Удаление файлов из редактора (Inline Media Deletion)**: на превью медиафайлов внутри редактора (в режиме Edit) добавлена плавающая кнопка удаления (красная корзина, видимая при наведении). По клику файл удаляется физически с сервера, а все ссылки на него в текущем документе (как `![[filename]]`, так и `![alt](assets/filename)`) автоматически стираются.
  * **Детекция неиспользуемых файлов (Unreferenced / Orphaned Files)**: на сервере реализовано быстрое сканирование текстов всех заметок на предмет упоминаний вложений. В панели управления медиафайлами появился фильтр **«Непривязанные»** со счетчиком неиспользуемых файлов, а на карточках неиспользуемых файлов отображается ярлык **«Не используется»**, что позволяет легко находить и удалять мусорные ассеты.
  * **Поддержка любых вложений в «Другие»**: серверный эндпоинт `/api/notes/media` теперь отдает все файлы из папки `assets/` (за исключением скрытых dot-файлов), благодаря чему файлы документов, аудиозаписи (.mp3/.wav), архивы (.zip) и PDF отображаются в библиотеке и фильтруются в категории **«Другие»**.
  * Загрузка медиа в редактор через Drag-and-Drop, вставку из буфера обмена (Ctrl+V) или диалог выбора. Автоименование скриншотов.
  * Централизованное хранилище в `assets/` с автоматическим разрешением конфликтов имен через числовые суффиксы `(1)`.
  * Защищенный GET-эндпоинт `/api/raw/*` с проверкой JWT и защитой от Directory Traversal.
  * **Оптимизация и миниатюры (Thumbnails)**:
    - **Изображения**: реализована динамическая генерация превью (миниатюр) шириной 300px с использованием библиотеки `sharp` и кэшированием в папке `_app/server/cache/thumbnails/`. Это снижает вес картинок в сетке на 93% и экономит сетевой трафик.
    - **Видео**: реализована ленивая подгрузка (Lazy Loading через `IntersectionObserver`) и автоматическое извлечение первого кадра в качестве превью на клиенте с помощью HTML5 Canvas (потребляет всего ~150-200 КБ трафика на видео вместо полной предзагрузки). При наведении курсора мыши запускается бесшумное превью-воспроизведение.
    - **Просмотр оригинала**: клик по любому превью (картинке или видео) открывает файл в исходном качестве в новой вкладке.

Доступна только пользователям с ролью `Admin` (для настроек) и всем авторизованным пользователям (для экспорта в боковом меню).

* **Регулировка ширины сайдбара (Desktop)**:
  - Реализовано перетаскивание правой границы сайдбара мышкой с ограничением размера от `200px` до `600px`. Выбранная ширина сохраняется в `localStorage` и автоматически восстанавливается при перезагрузке.

Сайдбар и тяжелый контент справа (включая схемы Mermaid) остаются неподвижными во время перетаскивания (0% нагрузки на процессор), а реальный сдвиг и релайаут происходят мгновенно за один кадр в момент отпускания мыши.
* **Полная локализация (Localization)**:
  - Панель управления администратора полностью переведена на русский и английский языки.
  - Все вкладки панели настроек («Общие», «Импорт & Загрузка», «Пользователи», «Мультимедиа», «Синхронизация», «О системе») и их содержимое (формы, заголовки, предупреждения, таблицы и логи) переключаются с помощью утилиты `t()` на основе выбранного в первой вкладке языка.

Панель настроек состоит из трех вкладок и панели экспорта:
1. **Импорт & Загрузка**:
   * **Поблочная загрузка (Chunked Upload)**: автоматическая нарезка ZIP-архивов на чанки по 5 МБ для стабильного импорта файлов объемом 300 МБ+ без тайм-аутов.
   * **Шкала прогресса (Progress Bar)**: отображение прогресса загрузки и фаз обработки на сервере.

   * Загрузка одиночных `.md` документов в выбранную папку.
   * **Экспорт хранилища**: гибкий выбор состава ZIP-архива (только заметки Markdown, только медиафайлы `assets/` или всё вместе) через модальное окно настроек экспорта (`ExportModal.tsx`). Название ZIP-архива генерируется динамически с префиксом типа архива (`obsidian-vault-notes-...`, `obsidian-vault-assets-...`, или `obsidian-vault-export-...`) и временной меткой в московском часовом поясе (MSK, UTC+3) независимо от региональных настроек сервера. Импортер при слиянии сохраняет существующие на сервере папки, позволяя загружать файлы по частям.

### Двусторонняя синхронизация локальной папки через MCP (`_sync_mcp/` + `routes/sync.js` + `SettingsPanel.tsx`)
* **Локальный MCP-агент с поддержкой WebSockets**: Разработано Node.js-приложение в папке `_sync_mcp/`, которое запускается на компьютере пользователя, реализует Model Context Protocol (MCP) и подключается по сокетам к серверу StrataNote. Это позволяет мгновенно получать сигналы об изменениях файлов на сервере и команды ручного управления.
* **Автоматизация установки и запуска**:
  * В команду пакетной установки `npm run install:all` добавлена автоустановка зависимостей в папку `_sync_mcp`.
  * При локальном запуске проекта (через `start.bat` или команду `npm run dev`) система автоматически проверяет наличие зависимостей в `_sync_mcp` и запускает локальный агент синхронизации параллельно с сервером бэкенда.
* **Фоновый Watcher (Chokidar)**: Встроенный в локальный агент watcher автоматически реагирует на изменения в локальной папке и с задержкой в 5 секунд отправляет обновленные файлы на сервер.
* **Умная логика Base-состояний**: Локальный агент хранит файл `.sync_state.json`, фиксирующий хэши файлов на момент последней успешной синхронизации. Это позволяет корректно отслеживать односторонние изменения и избегать ложных конфликтов.
* **Разрешение конфликтов**:
  * В случае одновременного изменения файла локально и на сервере, локальный агент отправляет свои изменения на сервер в виде предложения рецензирования (**Suggest Mode / Suggestion**). Локальный файл при этом защищен от затирания.
  * Также поддерживаются стратегии перезаписи: `local-wins` (локальная версия приоритетна), `server-wins` (серверная приоритетна, перед перезаписью локальный файл бэкапится в `.sync_backup/`) и `interactive` (создание конфликтных файлов рядом).
* **Интерактивное управление из админки (для ручного режима `manual`)**: В панели настроек StrataNote во вкладке **«Синхронизация»** для устройств, находящихся онлайн в ручном режиме, выводится кнопка **«Синхронизировать сейчас»**. При клике сервер отправляет команду по веб-сокету локальному агенту, агент запускает синхронизацию и возвращает логи ее выполнения прямо в веб-интерфейс админки.
* **Мониторинг в реальном времени**: В панели настроек StrataNote во вкладке **«Синхронизация»** отображается статус подключенного устройства (Подключен / Оффлайн / Ошибка), режим синхронизации (Авто / Ручной), время последней активности и текст ошибки, если она возникла. Также здесь реализована функция быстрого просмотра и копирования личного API-токена (JWT) для вставки в конфигурационный файл локального агента.
* **Таблица `sync_status` в БД SQLite**: Сервер бэкенда записывает статус синхронизации от каждого устройства для вывода в интерфейсе администратора.
* **Повышенная отказоустойчивость и производительность (Асинхронность, Кэш хешей, Чанки, Ретраи)**:
  * **Асинхронность и Event Loop**: Синхронное сканирование папок и хеширование файлов заменены на асинхронные стримы и итераторы с yield-механизмом (`setImmediate()`), предотвращая зависание веб-сокетов из-за блокировки Event Loop.
  * **Внутрипамятий кэш хешей (`fileHashCache`)**: Хеши файлов сохраняются в оперативной памяти и пересчитываются только при изменении размера или `mtime` файла. Это сократило время сканирования больших репозиториев (4600+ файлов) с 2 минут до 100 мс.
  * **Поблочный Push для больших файлов**: Файлы крупнее 1 МБ автоматически нарезаются агентом на чанки по 2 МБ и загружаются через новый эндпоинт `/api/sync/push-chunk`, решая проблему лимитов прокси-серверов (`413 Payload Too Large`).
  * **Автоматические повторные попытки (Retries)**: Все сетевые запросы при сбоях автоматически повторяются до 3 раз с экспоненциальной задержкой. Опции Socket.io клиента оптимизированы для надежного переподключения, а тайм-аут сокет-триггера на сервере увеличен с 45 секунд до 5 минут.

2. **Пользователи и Роли**:
   * Просмотр списка пользователей со статусом активности.
   * **Модерация регистраций**: новые пользователи ожидают активации администратором (ошибка `403` при входе до активации).
   * Запрет самостоятельной регистрации на роль `Admin` (блокируется на бэкенде).
   * Добавление, редактирование (имя, роль, активность, пароль) и удаление пользователей.
   * Защита от самоблокировки и самоудаления главного администратора.
3. **Мультимедиа**:
   * Сетка всех загруженных изображений и видео из `assets/` с указанием размера и даты изменения.
   * Удобный поиск по имени файла и **быстрые фильтры по категориям (Все, Изображения, Видео, Другие)** с автоматическим подсчетом количества файлов.
   * Физическое удаление файлов с диска с защитой от Directory Traversal.
4. **О системе (Changelog & Среда выполнения)**:

### Двусторонняя синхронизация и локальный MCP-агент (`_sync_mcp/`)
* **Локальный агент синхронизации (Sync Agent & MCP Server)**: встроен в папку `_sync_mcp/`. Позволяет пользователям работать с базой знаний локально (в Obsidian/VSCode) и автоматически или вручную синхронизировать её с веб-версией StrataNote.
* **Реал-тайм визуализация прогресса**: в панель настроек «Синхронизация» добавлен интерактивный Progress Bar со статусом текущего шага в реальном времени (отправка, получение, удаление файлов) через WebSocket-вещание `sync-agent-progress`. Исправлена видимость кнопки «Синхронизировать сейчас» и прогресс-бара — теперь они доступны для повторного ручного запуска в любых состояниях (если статус устройства не равен `offline`).
* **Исправление совместимости с Windows (Path Normalization)**: локальные относительные пути теперь принудительно нормализуются к Unix-формату с прямыми слэшами (`/`). Это решает проблему, когда файлы повторно копировались и дублировались в очередях отправки/получения из-за несовпадения символов слэша (`\` на Windows и `/` на сервере).
* **Защита от перезаписи и коллизий (Defensive Exclusions)**: 
  * Шаблоны исключений дополнены жестким списком системных папок (`_app`, `_sync_mcp`, `node_modules`, `.git`, `.obsidian`, `.agents`, `.sync_backup`) на сервере и клиенте. Любые пути, содержащие эти директории или скрытые файлы, полностью игнорируются при сканировании, что предотвращает случайное удаление исходного кода или системных каталогов.
  * Добавлена очистка базы данных на старте сервера и фильтрация в файловом наблюдателе (Chokidar), благодаря чему системные папки репозитория (например, `_sync_mcp`) автоматически удаляются из таблицы `notes` и больше не засоряют проводник заметок (Sidebar Explorer).
  * База метаданных `.sync_state.json` и бэкапы конфликтов `.sync_backup` перенесены внутрь каталога `_sync_mcp/`, очищая корень Obsidian от служебных файлов.
* **Исправление логов удаленного запуска**: изменен формат ответа WebSocket-коллбэка в `_sync_mcp/index.js` (убран лидирующий `null`), что позволило серверу корректно получать и отображать логи синхронизации в интерфейсе настроек админки.
* **Привязка автора изменений при синхронизации и редактировании**:
  * В эндпоинте `/api/sync/push` добавлено непосредственное сохранение версии заметки с именем авторизованного пользователя (`req.user.username`) перед записью файла на диск.
  * Добавлена двусторонняя синхронизация метаданных заметок (`created_by`, `last_edited_by` и всей истории версий из таблицы `versions`). Локальный агент `_sync_mcp` считывает метаданные из локальной БД SQLite и передает их в `/api/sync/push`, а при получении изменений от `/api/sync/pull` запрашивает метаданные с сервера и вносит их в локальную БД. Это полностью предотвращает затирание авторов и дат правок меткой "Внешняя система".
  * Исправлен импорт недостающей функции `basename` в файле `sync.js`, из-за отсутствия которой запись версий в БД при push падала с ошибкой, приводя к race condition с watcher-ом и назначению автора "Внешняя система".
  * В обработчике сохранения заметок `PUT /api/notes/` ([_app/server/routes/notes.js](file:///d:/YandexDisk/AWG/md_obsidian/_app/server/routes/notes.js)) порядок операций изменен так, что запись версии в SQLite происходит ДО физической записи на диск. Это полностью устраняет состояние гонки (race condition) с Chokidar-наблюдателем и гарантирует, что авторство изменений, сделанных в веб-интерфейсе локального или удаленного сервера, корректно закрепляется за вошедшим пользователем (например, `admin`), а не затирается значением `'Внешняя система'`.
* **Безопасность сокет-соединения и гибкое управление токенами**:
  * **Проверка подписи в WebSockets**: Введена обязательная валидация JWT-токена с помощью `jwt.verify(token, JWT_SECRET)` из handshake при регистрации локального агента (`register-sync-agent`). Если токен невалидный или истек, сервер отправляет событие `register-failed`, записывает статус ошибки в БД и принудительно отключает сокет, устраняя фантомное отображение устройства как «Подключен».
  * **Реал-тайм обновление присутствия**: Реализована автоматическая рассылка события `sync-status-changed` при любых изменениях статусов агентов (подключение, отключение, ошибка авторизации, завершение синхронизации). Фронтенд слушает это событие и обновляет список устройств на лету без ручного нажатия кнопки «Обновить». Исправлена ошибка, при которой статус устройства не сбрасывался в `offline` при отключении сокета после успешной синхронизации (теперь сброс происходит принудительно для любого предыдущего статуса, а также при старте сервера).
  * **Срок сессии 7 дней**: Стандартный сессионный токен, создаваемый при входе в систему, теперь действует **7 дней** (`expiresIn: '7d'`) вместо 24 часов.
  * **Генератор кастомных API-токенов**: Вкладка «Синхронизация» дополнена формой выбора периода действия токена (1 день, 7 дней, 30 дней, 90 дней, бессрочно/10 лет) и кнопкой **«Сгенерировать новый токен»** для удобного создания долгоживущих токенов локального агента.
  * **Дефолтный токен агента**: Вшит новый 10-летний токен по умолчанию, подписанный корректным ключом ребрендинга, обеспечивающий мгновенный локальный запуск.

### Режим рецензирования и предложений (Suggest Mode / Track Changes)
* **Создание и обновление предложений**: пользователи могут переключать редактор в режим рецензирования. Внесенные изменения отправляются на сервер и сохраняются как предложения в таблице `suggestions` базы данных. При этом пользователям разрешается редактировать контент локально, даже если файл заблокирован другим пользователем.
* **Разграничение прав на изменение (Вариант Б)**: пользователи с ролью `Editor` могут напрямую изменять только свои собственные файлы. Файлы «Внешней системы» и файлы, созданные другими пользователями или администраторами (включая `admin`), обычные редакторы редактировать напрямую не могут — для них доступен только Режим рецензирования (`Suggest Mode`). При попытке прямого редактирования бэкенд блокирует сохранение с ошибкой 403 и предлагает использовать режим предложений. В самом веб-интерфейсе редактора (при переходе к написанию кода) для таких файлов автоматически активируется Режим рецензирования, а кнопка его выключения блокируется в тулбаре с подсказкой о причинах.
* **Интерактивная система уведомлений (Realtime Notifications)**:
  * В левой панели (сайдбаре) рядом с аватаром пользователя интегрирована иконка колокольчика со счетчиком ожидающих предложений.
  * Счетчик мигает (пульсирует) в реальном времени при получении новых рецензий (через веб-сокеты).
  * Клик по колокольчику открывает всплывающий список предложений, направленных на согласование (содержит кнопку-крестик «X» для удобного закрытия поповера).
  * Клик на конкретное уведомление автоматически открывает нужный файл, активирует сайдбар рецензирования и запускает Diff-режим просмотра данного предложения.
  * Для `Admin` отображаются все активные предложения в системе. Для `Editor` — только предложения к файлам, автором которых он является.
* **Сравнение версий (Diff)**: в сайдбаре выводится список предложенных изменений от коллег. При клике на предложение открывается режим просмотра изменений с возможностью переключения между оригиналом, результатом и разницей (построчный Diff через `DiffViewer`).
* **Разрешение конфликтов (Merge Conflicts)**: при принятии правок выполняется автоматическое трехстороннее слияние (Three-Way Merge) между базовым контентом предложения, текущим содержимым на диске и предложенным. В случае конфликта (изменена одна и та же строка) бэкенд возвращает ошибку, а модератор получает удобный интерфейс слияния для ручного исправления строк и сохранения финальной версии.

## 4. Локальный запуск и тестирование

Для запуска и тестирования проекта на локальной машине выполните следующие шаги в терминале:

### Шаг 1. Установка зависимостей (выполняется один раз)
Перейдите в папку `_app` и запустите пакетную установку:
```bash
cd _app
npm run install:all
```

### Шаг 2. Запуск проекта
Вы можете выбрать один из двух режимов запуска:

#### А. Запуск собранной production-версии (Рекомендуется)
Этот режим запускает бэкенд, который раздает скомпилированные и оптимизированные файлы фронтенда статически с порта `3001`:
1. Убедитесь, что проект собран:
   ```bash
   cd _app
   npm run build
   ```
2. Запустите сервер:
   ```bash
   npm start
   ```
3. Откройте в браузере: **[http://127.0.0.1:3001](http://127.0.0.1:3001)**

#### Б. Режим разработки (с горячей перезагрузкой кода)
В этом режиме фронтенд-сервер Vite запускается отдельно на порту `5173`, отслеживая изменения в реальном времени, а запросы к API проксируются на бэкенд (порт по умолчанию `3001` или порт из переменной окружения `PORT`):
1. Запустите клиент и сервер одновременно из папки `_app`:
   ```bash
   cd _app
   npm run dev
   ```
2. Откройте в браузере: **[http://localhost:5173](http://localhost:5173)**

### Шаг 3. Быстрый запуск с автоматическим выбором порта (Windows)
Вы можете запустить проект одной кнопкой с помощью скрипта автоматического запуска:
1. Запустите файл `start.bat` в корневом каталоге проекта.
2. Скрипт сам перенаправит выполнение в `_app`, найдет свободный локальный порт (начиная с 3001), соберет клиентские файлы (если сборка отсутствует), запустит сервер на этом порту и автоматически откроет приложение в вашем браузере по умолчанию.
